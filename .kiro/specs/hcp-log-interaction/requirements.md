# Requirements Document

## Introduction

The HCP Log Interaction Screen is a life sciences CRM feature that allows pharmaceutical field representatives to record healthcare professional (HCP) touchpoints — visits, calls, emails, webinars, sample drops, and follow-ups — in a structured, compliant, and efficient manner.

The screen provides two equivalent input modes: a structured form mode for fast, field-validated data entry, and a conversational chat mode where the rep speaks naturally and an AI agent converts the input into CRM-ready structured interaction data. Both modes write into the same canonical interaction object. The feature is built on a React/TypeScript frontend, FastAPI backend, PostgreSQL database, and a LangGraph-powered agent service.

---

## Glossary

- **HCP**: Healthcare Professional — a doctor, nurse, pharmacist, or other licensed medical practitioner who is a target of the field representative's outreach.
- **Field Rep**: A pharmaceutical sales representative who records HCP interactions.
- **Interaction**: A single documented touchpoint between a Field Rep and an HCP (e.g., visit, call, email, webinar, sample drop).
- **Interaction Draft**: An auto-saved, incomplete interaction record not yet submitted for compliance review.
- **Canonical Interaction Object**: The single normalized data structure that holds all interaction data regardless of which input mode was used.
- **Form Mode**: A structured input mode presenting labeled fields for manual data entry.
- **Chat Mode**: A conversational input mode where the Field Rep types or speaks naturally, and the AI Agent extracts structured interaction data.
- **AI Agent**: The LangGraph-powered orchestration service responsible for extracting entities, validating fields, checking compliance, and suggesting next actions.
- **Extraction Confidence Score**: A numeric value (0.0–1.0) assigned by the AI Agent indicating certainty of a specific extracted entity.
- **Next Best Action (NBA)**: A CRM recommendation for the Field Rep's next engagement step with the HCP, generated by the AI Agent.
- **Compliance Flag**: A marker on an interaction record indicating potential regulatory or policy concerns (e.g., off-label product claims, adverse event mentions).
- **Adverse Event (AE)**: An undesirable medical occurrence reported by or about an HCP, subject to mandatory pharmacovigilance reporting.
- **Audit Log**: An immutable, time-stamped record of all create, edit, and submit actions performed on an interaction record.
- **Draft Save**: Persisting the current state of an interaction without submitting it for compliance review.
- **Submit**: The final action that marks an interaction as complete and routes it through compliance review.
- **LangGraph**: The multi-step agent orchestration framework used to coordinate AI tools in the backend.
- **RTK Query**: Redux Toolkit Query, used on the frontend for API state management and caching.
- **Pydantic**: Python data validation library used in the FastAPI backend.

## Requirements

### Requirement 1: HCP Context Panel

**User Story:** As a Field Rep, I want to see the HCP's profile and recent interaction history before logging a new interaction, so that I have the context needed to fill in accurate and relevant details.

#### Acceptance Criteria

1. WHEN the Log Interaction Screen is opened with a valid `hcp_id`, THE HCP_Context_Panel SHALL display the HCP's name, specialty, affiliation, preferred channel, and territory.
2. WHEN the Log Interaction Screen is opened with a valid `hcp_id`, THE HCP_Context_Panel SHALL display the HCP's three most recent interactions including date, type, and summary.
3. WHEN the Log Interaction Screen is opened with a valid `hcp_id`, THE HCP_Context_Panel SHALL display the products associated with the HCP's profile.
4. IF the `hcp_id` does not correspond to an existing HCP record, THEN THE HCP_Context_Panel SHALL display an error message and prevent the Field Rep from beginning a new interaction.
5. WHEN the Field Rep enters a search query of at least 2 characters into the HCP search field, THE HCP_Search_Component SHALL display a list of matching HCPs within 500ms.

---

### Requirement 2: Dual Input Mode — Form Mode

**User Story:** As a Field Rep, I want to enter interaction details using a structured form, so that I can quickly record a compliant interaction without relying on AI extraction.

#### Acceptance Criteria

1. THE Interaction_Form SHALL present the following fields: HCP ID (read-only), interaction type, interaction date and time, location or channel, products discussed, discussion topics, HCP sentiment, objections or barriers, samples provided, follow-up required (boolean), follow-up date, free-text notes, and compliance flag fields.
2. WHEN the Field Rep selects an interaction type from a predefined list, THE Interaction_Form SHALL update the required field set to match the rules for that interaction type.
3. WHEN the Field Rep sets "follow-up required" to true, THE Interaction_Form SHALL require a follow-up date before allowing submission.
4. WHEN the Field Rep submits the form with any required field empty, THE Interaction_Form SHALL display a field-level validation error message identifying each missing or invalid field.
5. THE Interaction_Form SHALL accept free-text notes up to 5,000 characters and display a character count indicator.
6. WHEN the Field Rep selects a date and time for the interaction, THE Interaction_Form SHALL validate that the date is not in the future and display an error if it is.

---

### Requirement 3: Dual Input Mode — Chat Mode

**User Story:** As a Field Rep, I want to describe my HCP interaction in natural language, so that the AI Agent can extract structured data and reduce my manual data entry burden.

#### Acceptance Criteria

1. WHEN the Field Rep switches to Chat Mode, THE Chat_Interface SHALL display a text input area, a send button, and a scrollable conversation history panel.
2. WHEN the Field Rep submits a natural language message, THE Chat_Interface SHALL display the message in the conversation history and show a loading indicator while the AI Agent processes the input.
3. WHEN the AI Agent returns a response, THE Chat_Interface SHALL display the AI summary, the extracted structured fields in an editable preview card, and a confidence indicator for each extracted field.
4. WHEN an extracted field has a confidence score below 0.75, THE Chat_Interface SHALL highlight that field and prompt the Field Rep to confirm or correct the value.
5. WHEN the AI Agent determines that a required field cannot be extracted from the input, THE Chat_Interface SHALL display a clarification prompt asking the Field Rep to provide the missing information.
6. WHEN the Field Rep edits an AI-extracted field in the preview card, THE Interaction_Draft_Store SHALL update the Canonical Interaction Object with the Field Rep's value and mark that field as manually overridden.
7. THE Chat_Interface SHALL preserve the full conversation history within an active session so the Field Rep can review the context of prior turns.

---

### Requirement 4: Unified Canonical Interaction Object

**User Story:** As a Field Rep, I want my interaction data to be consistent whether I use Form Mode or Chat Mode, so that I do not have to re-enter data when switching between modes.

#### Acceptance Criteria

1. THE Interaction_Draft_Store SHALL maintain a single Canonical Interaction Object shared across Form Mode and Chat Mode.
2. WHEN the Field Rep switches from Chat Mode to Form Mode, THE Interaction_Form SHALL pre-populate all fields with values currently held in the Canonical Interaction Object.
3. WHEN the Field Rep switches from Form Mode to Chat Mode, THE Chat_Interface SHALL reflect the current field values from the Canonical Interaction Object in the editable preview card.
4. WHEN a field is updated in either mode, THE Interaction_Draft_Store SHALL update the corresponding field in the Canonical Interaction Object within 200ms.

---

### Requirement 5: AI Agent — Entity Extraction and Orchestration

**User Story:** As a Field Rep, I want the AI Agent to extract interaction entities from my natural language input, so that structured CRM data is generated without manual entry.

#### Acceptance Criteria

1. WHEN the Field Rep submits a chat message, THE AI_Agent SHALL invoke the LangGraph workflow to extract the following entities: interaction type, date and time, channel or location, products discussed, discussion topics, HCP sentiment, objections or barriers, samples provided, follow-up required, and follow-up date.
2. WHEN entity extraction is complete, THE AI_Agent SHALL return an extraction confidence score between 0.0 and 1.0 for each extracted field.
3. WHEN the extraction confidence score for any required field is below 0.75, THE AI_Agent SHALL escalate processing to the llama-3.3-70b-versatile model before returning a response.
4. WHEN the extracted input indicates unusual complexity (more than 500 words) or compliance ambiguity, THE AI_Agent SHALL escalate processing to the llama-3.3-70b-versatile model.
5. WHEN entity extraction completes successfully, THE AI_Agent SHALL generate a concise human-readable summary of the interaction (maximum 200 words).
6. FOR ALL valid natural language inputs describing a complete interaction, THE AI_Agent SHALL return an extracted Canonical Interaction Object that, when re-serialized to natural language and re-extracted, produces an equivalent Canonical Interaction Object (round-trip equivalence property).

---

### Requirement 6: AI Agent — Compliance Check

**User Story:** As a Compliance Officer, I want the AI Agent to automatically detect compliance risks in interaction records, so that field reps are alerted before submitting problematic data.

#### Acceptance Criteria

1. WHEN the Field Rep initiates a submission or requests a compliance check, THE Compliance_Check_Tool SHALL scan the interaction record for the following risk categories: off-label product claims, prohibited promotional language, adverse event mentions, and missing mandatory fields.
2. WHEN a compliance risk is detected, THE Compliance_Check_Tool SHALL return a structured flag including: risk category, offending text or field, severity level (low, medium, high), and a recommended remediation action.
3. WHEN an adverse event mention is detected, THE Compliance_Check_Tool SHALL set the interaction's compliance flag to "AE Detected" and prevent final submission until the Field Rep acknowledges the pharmacovigilance reporting requirement.
4. WHEN no compliance risks are detected, THE Compliance_Check_Tool SHALL return a clean compliance status and allow the submission flow to proceed.
5. WHEN the compliance check is in progress, THE UI SHALL display a loading indicator and prevent the Field Rep from modifying the interaction record.

---

### Requirement 7: AI Agent — Next Best Action

**User Story:** As a Field Rep, I want the AI Agent to recommend a next engagement action after I log an interaction, so that I can maintain effective follow-up cadence with my HCPs.

#### Acceptance Criteria

1. WHEN an interaction record is submitted, THE Next_Best_Action_Tool SHALL analyze the interaction outcome, HCP interaction history, and territory goals to generate a recommended next action.
2. WHEN a next action is generated, THE AI_Agent SHALL return the action type (e.g., call, email, sample drop), a recommended timing window (e.g., within 7 days), and a brief rationale (maximum 100 words).
3. WHEN the Field Rep accepts the next best action recommendation, THE Follow_Up_Scheduler_Tool SHALL create a CRM follow-up task with the recommended action type, timing, and HCP identifier.
4. WHEN the Field Rep dismisses the next best action recommendation, THE Interaction_Draft_Store SHALL record the dismissal without creating a follow-up task.

---

### Requirement 8: Draft Save and Auto-Save

**User Story:** As a Field Rep, I want my in-progress interaction to be saved automatically, so that I do not lose data if I navigate away or experience an interruption.

#### Acceptance Criteria

1. WHILE an interaction is in progress, THE Interaction_Draft_Store SHALL auto-save the Canonical Interaction Object to the backend every 30 seconds.
2. WHEN the Field Rep explicitly clicks "Save Draft", THE Interaction_Draft_Store SHALL immediately persist the current Canonical Interaction Object and display a confirmation timestamp.
3. WHEN a draft save succeeds, THE Backend SHALL store the raw input (notes, chat turns), the AI interpretation layer (extracted entities, confidence scores, tool trace), and the current user-approved field values as three distinct data layers.
4. IF a draft save fails due to a network error, THEN THE Interaction_Draft_Store SHALL retain the unsaved state locally, display a non-blocking error notification, and retry the save within 10 seconds.
5. WHEN the Field Rep returns to the Log Interaction Screen for an HCP with an existing draft, THE Log_Interaction_Screen SHALL detect and offer to resume the existing draft.

---

### Requirement 9: Final Submission

**User Story:** As a Field Rep, I want to submit a completed interaction record after confirming its accuracy, so that it is persisted as the official CRM record subject to compliance review.

#### Acceptance Criteria

1. WHEN the Field Rep clicks "Submit", THE Log_Interaction_Screen SHALL display a confirmation dialog summarizing the interaction record before finalizing.
2. WHEN the Field Rep confirms submission, THE Submission_Handler SHALL run a compliance check and, if no blocking flags are present, persist the interaction with status "submitted".
3. WHEN submission succeeds, THE Log_Interaction_Screen SHALL display a success confirmation and navigate the Field Rep to the HCP's interaction history view.
4. IF the compliance check returns a high-severity flag, THEN THE Submission_Handler SHALL block submission, display the compliance flag details to the Field Rep, and require acknowledgment or correction before proceeding.
5. WHEN an interaction is submitted, THE Audit_Log SHALL record the Field Rep's user ID, the submission timestamp, the final field values, and the compliance check result as an immutable entry.

---

### Requirement 10: Edit Interaction via Natural Language

**User Story:** As a Field Rep, I want to modify a submitted or draft interaction using natural language instructions, so that I can make corrections efficiently without navigating individual form fields.

#### Acceptance Criteria

1. WHEN the Field Rep submits a natural language edit instruction, THE Edit_Interaction_Tool SHALL parse the instruction, identify the target field or fields, apply the change to the Canonical Interaction Object, and return the updated record.
2. WHEN an edit is applied, THE Audit_Log SHALL record the edit with: the original value, the new value, the natural language instruction, the timestamp, and the Field Rep's user ID.
3. WHEN an edit instruction is ambiguous or targets a field that cannot be identified, THE Edit_Interaction_Tool SHALL return a clarification prompt rather than applying a partial or incorrect change.
4. WHEN an edit modifies a field that was previously compliance-checked, THE Edit_Interaction_Tool SHALL invalidate the prior compliance check result and require a new compliance check before re-submission.

---

### Requirement 11: Audit Log

**User Story:** As a Compliance Officer, I want a complete, immutable audit trail for every interaction record, so that I can reconstruct the full history of how an interaction was created, edited, and submitted.

#### Acceptance Criteria

1. THE Audit_Log SHALL record an entry for each of the following events: interaction created, draft saved, field edited (form mode), AI extraction applied, AI extraction overridden by Field Rep, compliance check run, submission attempted, submission confirmed, and edit applied post-submission.
2. WHEN an audit log entry is written, THE Audit_Log SHALL include: event type, timestamp (UTC), user ID, interaction ID, and a JSON snapshot of the field values at the time of the event.
3. THE Audit_Log SHALL be append-only; no existing audit log entry SHALL be modifiable or deletable by any user role including administrators.
4. WHEN a Compliance Officer queries the audit log for a specific interaction, THE Audit_Log_API SHALL return all entries for that interaction ordered by timestamp ascending within 1 second.

---

### Requirement 12: Role-Based Access Control

**User Story:** As a system administrator, I want interaction records to be accessible only to authorized roles, so that sensitive HCP data is protected in accordance with data governance requirements.

#### Acceptance Criteria

1. WHEN a user attempts to access the Log Interaction Screen, THE Auth_Service SHALL verify that the user holds the "field_rep" or "manager" role before granting access.
2. WHEN a user with the "field_rep" role submits a query for interactions, THE Interaction_API SHALL return only interaction records associated with that Field Rep's territory.
3. WHEN a user with the "compliance_officer" role submits a query for interactions, THE Interaction_API SHALL return interaction records across all territories that contain a compliance flag.
4. IF a user without a valid session token attempts to call any backend API endpoint, THEN THE Auth_Service SHALL return an HTTP 401 response and log the unauthorized access attempt.

---

### Requirement 13: Performance and Availability

**User Story:** As a Field Rep, I want the Log Interaction Screen to load quickly and respond without noticeable delay, so that I can complete interaction logging efficiently in the field.

#### Acceptance Criteria

1. WHEN the Log Interaction Screen is opened, THE Frontend SHALL render the HCP context panel and the active input mode within 2 seconds on a standard 4G mobile connection.
2. WHEN the Field Rep submits a chat message for AI extraction, THE AI_Agent SHALL return extracted entities and a summary within 5 seconds for inputs under 500 words using the gemma2-9b-it model.
3. WHEN the AI_Agent escalates to the llama-3.3-70b-versatile model, THE AI_Agent SHALL return a response within 15 seconds.
4. WHILE the AI_Agent is processing a request, THE Chat_Interface SHALL display a loading indicator and remain interactive for non-AI actions (e.g., switching modes, editing previously extracted fields).
5. WHEN a draft save API call is made, THE Backend SHALL persist the draft and return a success response within 1 second under normal load conditions.
